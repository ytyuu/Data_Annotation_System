# 大模型标注失败批量重试设计

## 1. 文档状态

本文档定义大模型标注「失败原因记录 + 失败结果批量重试」能力的第一版方案，作为后续数据库、后端、Worker 和前端实现的共同基线。

当前状态：**仅完成设计，尚未开始编码和数据库变更。**

执行边界：

- Codex 负责设计、代码、SQL 文件、静态检查和不访问数据库的自动化测试。
- 用户负责执行数据库变更 SQL，并自行检查数据库结构和数据。
- Codex 不连接数据库，不执行 SQL，不查询数据库状态或业务数据。

## 2. 背景

当前系统已有三层有限的失败处理能力：

1. Worker 对模型小批请求执行 `maxRetries` 重试，默认额外重试 2 次。
2. 小批持续失败时递归拆分，单条仍失败后上传为 `failed`。
3. Worker 领取后异常退出时，5 分钟租约到期的数据可以在后续运行中重新领取，领取次数受批次 `maxAttempts` 限制。

当前缺口：

- 单条结果进入 `failed` 后不会被 Worker 自动重新领取。
- 现有 `error_message` 只保留最新错误，重试会覆盖或清空历史原因。
- 提供方只能逐条点击「重新标注」，没有选择失败项或重试全部失败项的能力。
- 「重新标注」只重置结果，批次已结束时不会自动重新派发 Worker。
- 整批 Worker 异常会将批次标记为 `failed`，当前运行接口又只允许 `pending`，无法形成恢复闭环。
- 错误只有文本，无法稳定区分模型超时、限流、输出校验失败、租约耗尽和 Worker 运行时异常。

## 3. 本期目标

本期必须形成以下闭环：

```text
结果或批次执行失败
  -> 保存结构化失败原因和当次失败快照
  -> 提供方查看失败原因
  -> 选择部分失败项或全部可重试失败项
  -> 确认批量重试
  -> 后端事务内校验、重新锁定数据项、重置结果
  -> 批次回到 pending 并自动派发 Worker
  -> Worker 重新处理
  -> 新失败继续追加历史，旧失败记录不丢失
```

必须完成：

- 记录结果级和批次级终态失败原因。
- 保留每次失败历史，不因重试覆盖。
- 支持「重试选中失败项」和「重试全部可重试失败项」。
- 重试操作具有 provider 归属校验、并发保护和请求幂等。
- 重试成功后自动派发 Worker；派发失败时保留可再次开始的 `pending` 状态。
- 页面展示失败原因、失败阶段、失败时间、领取次数和重试轮次。

## 4. 非目标

- 自动无限重试所有失败。
- 为不同模型提供复杂的动态退避策略管理页面。
- 自动重试鉴权错误、标注结构错误等明确不可恢复错误。
- 跨批次迁移失败结果。
- 修改已经接受、已转人工或已由人工流程处理的数据。
- 引入 Redis、消息队列或新的调度基础设施。
- 保存 API Key、Authorization Header、完整请求头或包含密钥的异常对象。

## 5. 核心概念

### 5.1 Worker 内部重试

一次运行内对模型请求的短暂重试。它不改变结果的 `retry_count`，只累计模型请求用量。默认沿用现有 `maxRetries = 2`。

### 5.2 领取尝试

结果被 Worker 领取一次，`attempt_count + 1`。租约超时后重新领取也算一次。批次配置 `maxAttempts` 约束单个重试轮次内的最大领取次数。

### 5.3 业务重试轮次

提供方对终态失败结果执行一次「重新标注」或「批量重试」，`retry_count + 1`，并将 `attempt_count` 重置为 0。

三者必须分开计数，避免把模型 HTTP 重试、租约重领和人工发起的业务重试混为一谈。

## 6. 失败分类

### 6.1 失败阶段

第一版使用受控字符串，不使用数据库枚举：

| `failure_stage` | 含义 | 作用域 |
|---|---|---|
| `model_request` | 模型网络、超时、限流或服务端失败 | result |
| `model_output` | 空响应、JSON 解析或结果结构校验失败 | result |
| `lease` | 领取次数达到上限 | result |
| `worker_runtime` | Worker 未捕获异常或进程级失败 | batch |
| `backend_request` | Worker 调用领取或上传接口失败 | batch |
| `batch_dispatch` | 后端无法派发 Worker | batch |
| `unknown` | 旧数据或无法可靠分类的错误 | result / batch |

### 6.2 失败代码

`failure_code` 使用稳定机器值，页面再映射中文文案。首批代码：

- `model_timeout`
- `model_rate_limited`
- `model_server_error`
- `model_auth_error`
- `empty_model_response`
- `invalid_json_output`
- `invalid_annotation_output`
- `lease_attempts_exhausted`
- `worker_runtime_error`
- `backend_unavailable`
- `result_upload_failed`
- `legacy_failure`
- `unknown_error`

### 6.3 是否可重试

后端保存 `retryable`，不由前端自行推断。

默认可重试：

- 超时、限流、模型 `5xx`。
- 空响应、非法 JSON、结构校验失败。
- 租约次数耗尽。
- Worker 运行时异常和后端暂时不可达导致的批次失败。
- 迁移前已有的 `legacy_failure`。

默认不可重试：

- 模型鉴权失败。
- 数据集标注结构无效。
- 不支持的内容类型。
- 数据项已进入人工任务、已接受或已经完成其他业务流转。

不可重试失败仍可查看和转人工，但不进入批量重试选择范围。

## 7. 数据模型

### 7.1 `ai_annotation_results` 增量字段

新增：

| 字段 | 类型 | 说明 |
|---|---|---|
| `failure_code` | `varchar(64) NULL` | 当前最后一次失败代码 |
| `failure_stage` | `varchar(32) NULL` | 当前最后一次失败阶段 |
| `failed_at` | `timestamptz NULL` | 当前最后一次失败时间 |
| `retryable` | `boolean NOT NULL DEFAULT false` | 当前失败是否允许业务重试 |
| `retry_count` | `int NOT NULL DEFAULT 0` | 已执行的业务重试轮次 |

保留现有 `error_message` 作为当前失败摘要，列表接口无需查询历史表即可展示。

重试时清空当前失败字段，但历史表记录保持不变：

```text
status -> pending
failure_code / failure_stage / failed_at / error_message -> NULL
retryable -> false
attempt_count -> 0
retry_count -> retry_count + 1
租约、结果、置信度、风险标记、审核字段 -> 清空
```

### 7.2 `ai_annotation_batches` 增量字段

新增：

| 字段 | 类型 | 说明 |
|---|---|---|
| `failure_code` | `varchar(64) NULL` | 当前批次失败代码 |
| `failure_stage` | `varchar(32) NULL` | 当前批次失败阶段 |
| `failed_at` | `timestamptz NULL` | 当前批次失败时间 |
| `retry_count` | `int NOT NULL DEFAULT 0` | 批次被恢复执行的次数 |

重试准备成功后，批次回到 `pending`，清空当前失败字段、`finished_at` 和 `error_message`，保留历史统计用量并刷新结果计数。

### 7.3 新表 `ai_annotation_failure_records`

追加式失败历史。失败内容不覆盖，仅允许在后续重试时补写 `retry_request_id`、`retried_by` 和 `retried_at` 解决信息：

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | `uuid PK` | 失败记录 ID |
| `batch_id` | `uuid NOT NULL` | 所属批次，批次删除时级联删除 |
| `result_id` | `uuid NULL` | 单条结果失败时填写 |
| `dataset_id` | `uuid NOT NULL` | 便于归属和审计查询 |
| `item_id` | `uuid NULL` | 单条结果失败时填写 |
| `scope` | `varchar(16)` | `result` 或 `batch` |
| `failure_stage` | `varchar(32)` | 失败阶段 |
| `failure_code` | `varchar(64)` | 稳定失败代码 |
| `error_message` | `text` | 对用户可展示的错误摘要，最多 2000 字符 |
| `retryable` | `boolean` | 失败发生时的可重试判断 |
| `attempt_count` | `int` | 失败时的领取次数快照 |
| `retry_count` | `int` | 失败时的业务重试轮次快照 |
| `raw_output` | `jsonb NULL` | 输出校验失败时保存模型原始输出快照 |
| `details` | `jsonb` | HTTP 状态、模型名、chunkNo 等脱敏信息 |
| `retry_request_id` | `uuid NULL` | 后续由哪次批量重试处理 |
| `retried_by` | `uuid NULL` | 发起重试的 provider |
| `retried_at` | `timestamptz NULL` | 进入新一轮处理的时间 |
| `created_at` | `timestamptz` | 失败发生时间 |

索引至少包括：

- `(batch_id, created_at DESC)`
- `(result_id, created_at DESC)`
- `(batch_id, retryable, retried_at)`
- `(retry_request_id)`

### 7.4 新表 `ai_annotation_retry_requests`

记录一次批量重试操作，并作为幂等边界：

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | `uuid PK` | 前端生成的 `requestId` |
| `batch_id` | `uuid NOT NULL` | 重试批次 |
| `provider_id` | `uuid NOT NULL` | 发起人 |
| `scope` | `varchar(24)` | `selected` 或 `all_failed` |
| `requested_count` | `int` | 请求数量 |
| `retried_count` | `int` | 实际重置数量 |
| `skipped_count` | `int` | 因状态变化跳过数量 |
| `status` | `varchar(24)` | `prepared`、`dispatched`、`dispatch_failed` |
| `comment` | `text NULL` | 重试说明 |
| `error_message` | `text NULL` | 派发失败原因 |
| `created_at` | `timestamptz` | 创建时间 |
| `updated_at` | `timestamptz` | 更新时间 |

同一个 `requestId` 重复提交时返回第一次操作结果，不再次重置结果或增加 `retry_count`。

### 7.5 旧失败数据

数据库变更 SQL 应为现有 `failed` 结果和批次补建一条 `legacy_failure` 历史记录：

- 沿用已有 `error_message`。
- `failure_stage = unknown`。
- `retryable = true`，允许提供方决定是否重试。
- 不修改非失败记录。

## 8. 可重试资格

结果必须同时满足：

1. 批次属于当前 provider。
2. 结果属于目标批次且 `status = failed`。
3. 当前失败 `retryable = true`。
4. 批次状态为 `completed` 或 `failed`。
5. 数据项满足以下一种状态：
   - `ai_processing`，且 `current_round_no` 与结果 `round_no` 一致；
   - `pending`，且没有人工任务或其他 AI 批次占用，可以重新原子锁定为 `ai_processing`。
6. 数据项不是 `accepted`、`annotated`、`assigned`、`disputed`、`rejected` 或其他已进入后续流程的状态。

并发规则：

- `selected`：请求中的任一结果不符合资格，整个请求返回 `409`，不做部分修改。
- `all_failed`：事务内锁定全部候选，只重试仍符合资格的结果，状态刚刚变化的项目计入 `skippedCount`。
- `running` 批次不允许发起批量重试，避免和正在执行的 Worker 竞争。

## 9. API 设计

### 9.1 批量重试

```http
POST /api/provider/ai-annotation-batches/{batchId}/retry-failed
Authorization: Bearer <provider JWT>
Content-Type: application/json
```

请求：

```json
{
  "requestId": "UUID",
  "scope": "selected",
  "resultIds": ["UUID"],
  "comment": "修复模型服务后重新执行"
}
```

规则：

- `scope = selected` 时，`resultIds` 必填，数量 1 至 500。
- `scope = all_failed` 时，`resultIds` 必须为空。
- `comment` 可选，最多 1000 字符。
- `all_failed` 最多处理当前批次允许的 5000 条结果，不受前端 UUID 列表长度限制。

成功派发返回 `202`：

```json
{
  "message": "失败结果已重置并派发",
  "requestId": "UUID",
  "retriedCount": 42,
  "skippedCount": 0,
  "batchStatus": "pending",
  "dispatchStatus": "dispatched"
}
```

重置成功但 Worker 不可达时返回 `200`，批次保持可开始的 `pending`：

```json
{
  "message": "失败结果已重置，Worker 暂不可达，请稍后开始批次",
  "requestId": "UUID",
  "retriedCount": 42,
  "skippedCount": 0,
  "batchStatus": "pending",
  "dispatchStatus": "dispatch_failed"
}
```

### 9.2 失败历史

```http
GET /api/provider/ai-annotation-results/{resultId}/failures?page=1&pageSize=50
```

仅返回当前 provider 所属结果的失败历史，默认按 `created_at DESC` 排序。

### 9.3 结果列表扩展

`AiAnnotationResultResponse` 新增：

- `failureCode`
- `failureStage`
- `failedAt`
- `retryable`
- `retryCount`

批次响应增加同名批次级失败字段和 `retryCount`。

## 10. 后端事务流程

批量重试必须先完成数据库事务，再派发 Worker：

1. 校验 provider、批次状态和请求幂等键。
2. 锁定批次和候选结果行。
3. 按 `selected` 或 `all_failed` 规则校验资格。
4. 对 `pending` 数据项执行带状态条件的原子更新，重新标记为 `ai_processing`。
5. 为将要重试的失败历史写入 `retry_request_id`、`retried_by`、`retried_at`。
6. 重置结果字段，`attempt_count = 0`，`retry_count + 1`。
7. 将批次置为 `pending`，`retry_count + 1`，清空当前批次错误和完成时间。
8. 刷新批次计数并写入 `ai_annotation_retry_requests(status = prepared)`。
9. 提交事务。
10. 调用现有 `AiWorkerDispatcher` 派发。
11. 更新重试请求为 `dispatched` 或 `dispatch_failed`。

派发失败不得回滚已经完成的结果重置，因为外部 HTTP 调用不能放入数据库事务；批次保持 `pending`，用户仍可使用现有开始入口再次执行。

## 11. Worker 改造

### 11.1 结构化失败上传

单条失败上传字段扩展为：

```json
{
  "errorCode": "model_timeout",
  "failureStage": "model_request",
  "errorMessage": "模型请求超时",
  "retryable": true,
  "rawOutput": null
}
```

Worker 负责把 SDK 异常映射为稳定代码，但后端仍需校验允许值，未知代码统一归一化为 `unknown_error`。

### 11.2 重试层次校准

- 保留 Worker 小批级指数退避。
- 明确设置 OpenAI SDK `maxRetries`，避免 SDK 默认重试与 Worker 外层重试叠加失控。
- `400`、`401`、`403` 和不可恢复配置错误不执行 Worker 外层重试。
- `408`、`409`、`429`、`5xx`、连接失败和超时允许重试。
- 输出为空、JSON 非法、结构校验失败允许按现有拆分策略重试。
- Worker 调用后端领取和上传接口应对连接错误、`408`、`429`、`5xx` 增加有限重试；上传继续依赖 `requestId` 保证幂等。

## 12. 前端体验

### 12.1 失败分类列表

失败 Tab 增加：

- 行选择框。
- 「重试选中（N）」按钮。
- 「重试全部失败（N）」按钮。
- 失败原因摘要、失败阶段、失败时间、领取次数、业务重试次数。
- 「查看失败历史」入口。

不可重试项目禁用选择框，并显示不可重试原因。

### 12.2 确认弹窗

确认弹窗必须显示：

- 批次名称和模型。
- 计划重试数量。
- 本次操作会重新调用模型并产生新的模型用量。
- 派发失败时结果仍会保持为可再次开始的状态。

确认按钮文案：`确认重试`。请求中禁用关闭和重复提交。

### 12.3 操作后反馈

- 派发成功：关闭弹窗，轮询批次，切换到最新批次进度。
- 派发失败：显示警告，批次呈现 `pending`，保留右侧播放开始入口。
- 没有可重试项：隐藏批量按钮，展示明确空状态。
- 重试完成后，旧失败记录仍可在历史中查看。

## 13. 安全与审计

- 所有重试和历史接口只允许数据集 provider 访问。
- 失败详情不得保存 API Key、Bearer Token、完整请求头和环境变量。
- `error_message` 对 SDK 异常进行清洗并限制 2000 字符。
- `details` 只允许白名单字段，例如 HTTP 状态、模型名、chunkNo、requestId 和错误类型。
- `raw_output` 仅保存模型响应，不保存完整 Prompt；沿用数据集访问权限。
- 重试请求永久记录发起人、时间、范围、数量和说明。

退出条件：用户确认 SQL 已执行，后端映射通过静态编译。

- [ ] Worker 增加错误分类器和结构化失败上传。

- [ ] 校准 SDK 与 Worker 两层重试次数。

卡点：必须区分可恢复错误和永久错误，禁止对鉴权错误进行递归拆分重试。

退出条件：每类终态失败都有稳定代码、中文展示文案和历史记录。

退出条件：重复请求不重复重置；并发请求不会让同一结果增加两次 `retry_count`。

## 15. 第一版验收标准

1. 一个批次存在多个 `failed` 结果时，可以选择部分或一次重试全部可重试结果。
2. 重试前后都能查看旧失败原因，新失败以新记录追加。
3. 同一 `requestId` 重复提交不会重复增加重试次数。
4. 已接受、已转人工、人工处理中或不可重试结果不会被重新领取。
5. 重试后批次自动派发；Worker 不可达时批次保持 `pending`，可以手动开始。
6. 批次级 Worker 异常有独立失败记录，并可通过批量重试恢复其中符合资格的结果。
7. 页面准确展示失败数量、失败原因、失败时间、领取次数和业务重试次数。
8. Worker 不对鉴权和配置错误进行无意义的递归拆分重试。
9. 任何日志和失败记录中均不出现密钥或 Authorization Header。

## 16. 已采用的默认决策

- 第一版选中重试上限为 500 条，重试全部上限与批次上限一致，为 5000 条。
- 仅 `completed` 和 `failed` 批次允许批量重试。
- 批量重试默认自动派发 Worker。
- 派发失败不回滚重置，批次保留为 `pending`。
- 历史只记录终态失败，不记录每一次短暂模型 HTTP 重试，避免记录量失控。
- 旧失败数据按 `legacy_failure` 迁移并默认允许重试。
- 当前错误保存在批次/结果主表用于列表展示，完整历史保存在追加式失败记录表。
