# 大模型标注失败批量重试实现步骤

## 1. 使用方式

本文档是[大模型标注失败批量重试设计](./大模型标注失败批量重试设计.md)的执行计划。后续实现必须按阶段顺序推进，每完成一个检查项就更新状态。

状态约定：

- `[ ]`：尚未开始。
- `[~]`：正在进行。
- `[x]`：已经完成并通过本阶段检查。
- `[!]`：遇到阻塞，必须记录原因和处理结论。

执行规则：

1. 每次只允许一个主要阶段处于 `[~]`。
2. 开始一个阶段前，确认依赖阶段已经达到退出条件。
3. 代码写完不等于阶段完成，必须执行本阶段验证。
4. 关键设计变化先更新设计方案，再同步修改本文档。
5. 不连接数据库，不执行 SQL，不查询数据库结构或业务数据。
6. 数据库变更必须由用户执行并明确确认后，才能继续依赖新结构的代码实现。
7. 不调用真实 DeepSeek，不使用真实业务批次做自动化验证。
8. 保持现有 CLI 与前端触发 Worker 两种运行方式兼容。

## 2. 当前执行总览

- `[x]` 阶段 0：设计确认与实现边界固化。
- `[~]` 阶段 1：数据库变更 SQL，由用户执行并确认。
- `[ ]` 阶段 2：数据库映射、领域模型和失败分类基础设施。
- `[ ]` 阶段 3：Worker 结构化失败与重试策略校准。
- `[ ]` 阶段 4：后端失败记录、失败历史和查询接口。
- `[ ]` 阶段 5：批量重试事务、幂等和 Worker 派发。
- `[ ]` 阶段 6：前端失败列表、批量重试和历史查看。
- `[ ]` 阶段 7：自动化测试、构建和文档校准。
- `[ ]` 阶段 8：用户端数据库与真实 Worker 联调。

## 3. 依赖关系

```text
阶段 0 设计确认
  -> 阶段 1 数据库变更
      -> 阶段 2 映射与领域基础
          -> 阶段 3 Worker 结构化失败
          -> 阶段 4 后端失败记录
              -> 阶段 5 批量重试与派发
                  -> 阶段 6 前端交互
                      -> 阶段 7 自动化验证
                          -> 阶段 8 真实联调
```

阶段 3 和阶段 4 的部分代码可以并行设计，但正式合并前必须以阶段 2 的失败代码、字段和请求模型为共同契约。

### 目标

确认第一版失败批量重试的业务边界，避免数据库完成后再修改核心状态机。

- `[x]` 编写[大模型标注失败批量重试设计](./大模型标注失败批量重试设计.md)。

- `[x]` 定义结果级与批次级失败阶段、失败代码和可重试判断。
- `[x]` 定义选中重试、全部重试、幂等请求和派发失败恢复方案。

### 待确认

- `[x]` 确认新增 `ai_annotation_failure_records` 和 `ai_annotation_retry_requests` 两张表。
- `[x]` 确认选中重试最多 500 条、全部重试最多 5000 条。
- `[x]` 确认仅允许 `completed` 和 `failed` 批次发起批量重试。
- `[x]` 确认重试准备完成后自动派发 Worker。
- `[x]` 确认派发失败不回滚结果重置，批次保持 `pending`。

### 卡点

- 如果失败历史只保留当前原因，可以减少一张表，但会失去重试前后的审计链路，不符合本期“失败原因记录”目标。
- 如果允许 `running` 批次重试，需要解决正在运行的 Worker 与重试事务竞争，本期默认不支持。

### 退出条件

- 用户确认设计方案中的默认决策，或明确给出调整项。
- 设计方案和本文档不存在相互冲突的字段、状态和接口定义。

## 7. 阶段 3：Worker 结构化失败与重试策略校准

### 目标

让 Worker 输出稳定失败原因，并避免 OpenAI SDK 与 Worker 外层重试叠加失控。

### 实现任务

- `[ ]` 新增 Worker 错误分类器，将 SDK、HTTP、超时和输出校验异常映射为结构化失败。
- `[ ]` 区分可重试异常与不可重试异常。
- `[ ]` 明确设置 OpenAI SDK `maxRetries`，不继续依赖默认值。
- `[ ]` 保留 Worker 小批级指数退避和随机抖动。
- `[ ]` `400`、`401`、`403` 和配置错误不执行外层重复请求。
- `[ ]` `408`、`409`、`429`、`5xx`、连接失败和超时允许有限重试。
- `[ ]` 空响应、非法 JSON 和结构校验失败继续支持拆分重试。
- `[ ]` 不可重试错误禁止递归拆分，直接生成终态失败。
- `[ ]` 单条终态失败上传 `errorCode`、`failureStage`、`errorMessage`、`retryable` 和可选 `rawOutput`。
- `[ ]` 批次级异常向 `/fail` 上报结构化失败字段。
- `[ ]` 为 Worker 调用后端的领取和上传增加有限网络重试。
- `[ ]` 上传重试沿用同一个 `requestId`，保证幂等。
- `[ ]` 日志增加失败代码、阶段和 Worker 尝试次数，不打印密钥或完整请求头。

### 验证任务

- `[ ]` 模型超时重试次数测试。
- `[ ]` `429`、`5xx` 退避测试。
- `[ ]` `401` 不重试、不拆分测试。
- `[ ]` 非法 JSON 拆分到单条并上传结构化失败测试。
- `[ ]` 上传失败使用相同 `requestId` 重试测试。
- `[ ]` Worker 全量测试和类型检查通过。

### 卡点

- 当前 `processWithFallback` 捕获所有异常并递归拆分，必须先分类再决定重试和拆分。
- SDK 内部重试与 Worker 外层重试必须只保留一套主要策略，否则永久错误会产生大量请求。
- 后端上传重试不能生成新的 `requestId`，否则可能重复累计模型用量。

### 退出条件

- 所有终态失败都有结构化代码。
- 永久错误不会触发指数级拆分请求。
- Worker 测试通过且日志中无敏感信息。

## 8. 阶段 4：后端失败记录、失败历史和查询接口

### 实现任务

- `[ ]` 新增失败记录 Store，统一写入结果级和批次级历史。
- `[ ]` 结果上传失败分支同时更新主表当前失败字段并追加历史。
- `[ ]` 租约领取次数耗尽时为每条结果追加 `lease_attempts_exhausted` 历史。

- `[ ]` 派发失败记录 `batch_dispatch`，但不把已准备重试的批次改为 `failed`。
- `[ ]` 成功结果清空结果主表当前失败字段，不删除历史。
- `[ ]` 批次成功进入运行或完成状态时清空批次当前失败字段，不删除历史。
- `[ ]` 扩展 `AiAnnotationResultResponse` 和 `AiAnnotationBatchResponse`。
- `[ ]` 新增失败历史分页响应模型。
- `[ ]` 新增 provider 失败历史查询路由。
- `[ ]` 查询时校验结果、批次和 provider 归属。
- `[ ]` `details` 和 `rawOutput` 按权限返回，不暴露敏感字段。

### 验证任务

- `[ ]` 失败归一化与历史记录构建单元测试。
- `[ ]` 结果成功后清空当前失败但保留历史的 Store 行为测试或静态验证。
- `[ ]` provider 越权查询失败历史的服务测试。
- `[ ]` 分页参数和排序测试。
- `[ ]` 后端测试与编译通过。

### 卡点

- 当前租约耗尽使用批量 `UPDATE`，实现历史记录时需要先锁定并取得结果快照，不能只更新状态。
- 当前批次失败会批量标记多个结果失败，需要避免为同一结果重复写入相同终态失败。
- 失败历史写入和主表状态更新必须处于同一事务。

### 退出条件

- 单条模型失败、租约耗尽和批次失败都有历史记录。
- 列表能展示最新原因，详情能查询完整历史。
- 后端测试和编译通过。

## 9. 阶段 5：批量重试事务、幂等和 Worker 派发

### 目标

实现可靠的选中重试、全部重试和自动派发，补齐现有「重新标注」无法唤醒 Worker 的缺口。

### 实现任务

- `[ ]` 新增 `RetryFailedAiAnnotationResultsRequest` 和响应模型。
- `[ ]` 新增 `POST /api/provider/ai-annotation-batches/{batchId}/retry-failed`。
- `[ ]` 校验 `requestId`、`scope`、结果数量和 comment 长度。
- `[ ]` 使用 `ai_annotation_retry_requests.id` 实现请求幂等。
- `[ ]` 锁定批次，限制状态为 `completed` 或 `failed`。
- `[ ]` `selected` 模式校验全部结果，任一不合格则事务不修改。
- `[ ]` `all_failed` 模式锁定全部可重试候选并统计跳过数量。
- `[ ]` 校验数据项状态和 `current_round_no`。
- `[ ]` 对可恢复的 `pending` 数据项执行带状态条件的原子锁定。
- `[ ]` 禁止重新占用人工任务中、已接受或已进入后续流程的数据项。
- `[ ]` 为失败历史补写重试请求、发起人和重试时间。
- `[ ]` 重置结果、清空租约和审核字段、`attempt_count = 0`、`retry_count + 1`。
- `[ ]` 批次回到 `pending`，清空当前错误和完成时间，`retry_count + 1`。
- `[ ]` 刷新批次统计并写入重试请求 `prepared` 状态。
- `[ ]` 事务提交后调用现有 `AiWorkerDispatcher`。
- `[ ]` 派发成功更新为 `dispatched` 并返回 `202`。
- `[ ]` 派发失败更新为 `dispatch_failed`，批次保持 `pending` 并返回可恢复响应。
- `[ ]` 将现有单条 `reject_retry` 收敛到相同重试准备与派发逻辑，避免产生无人执行的 `pending` 结果。

### 验证任务

- `[ ]` 重复 `requestId` 返回第一次结果，不重复增加 `retry_count`。
- `[ ]` `selected` 混入无效结果时全部回滚。
- `[ ]` `all_failed` 正确统计重试和跳过数量。
- `[ ]` `running`、`pending`、`cancelled` 批次拒绝批量重试。
- `[ ]` 已转人工或已接受数据项不会被重新锁定。
- `[ ]` 派发成功、Worker 不可达和 Worker 返回 `409` 的结果测试。
- `[ ]` 单条「重新标注」会触发 Worker 或留下可手动开始的 `pending` 批次。
- `[ ]` 后端测试与编译通过。

### 卡点

- 数据库事务与 Worker HTTP 派发不能组成原子事务，必须接受“已准备但未派发”的中间状态。
- 结果重置和数据项重新锁定必须使用条件更新，不能先查询后无条件写入。
- 幂等命中时不能再次派发 Worker；只返回已有派发状态。
- Worker 已经运行但后端状态尚未刷新时可能返回 `409`，需要映射成明确响应而不是重复重置。

### 退出条件

- 批量重试具备归属校验、资格校验、事务、幂等和派发闭环。
- 单条重试与批量重试共享核心状态逻辑。
- 后端自动化检查通过。

- 当前低风险和失败结果共用列表组件结构，选择状态必须按 Tab 隔离。
- 失败历史可能包含较长错误和原始输出，需要限制弹窗高度并允许内容区滚动。
- `dispatch_failed` 是“重置成功、自动启动失败”，不能显示成整个重试操作完全失败。

### 退出条件

- 提供方可以查看原因、选择、确认、批量重试并观察执行。
- 派发失败时有明确恢复入口。
- 前端构建通过并完成响应式检查。

## 12. 阶段 8：用户端数据库与真实 Worker 联调

### 目标

由用户在开发环境验证真实数据库事务和 Worker 执行闭环。

### 用户验证步骤

- `[ ]` 创建一个可控的小批次并开始执行。

- `[ ]` 选择部分失败项执行批量重试。

- `[ ]` 对剩余失败项执行重试全部。
- `[ ]` 重复提交同一 `requestId`，确认没有重复重置。
- `[ ]` 临时停止 Worker，执行批量重试，确认批次保持 `pending`。
- `[ ]` 恢复 Worker 后点击播放开始入口，确认继续执行。
- `[ ]` 制造批次级异常，确认批次失败原因和恢复流程。
- `[ ]` 验证已接受、已转人工和人工任务中的数据不会被重新领取。

### Codex 支持

- `[ ]` 根据用户提供的日志分析失败，不连接数据库。

### 退出条件

## 14. 完成定义

只有同时满足以下条件，才能将本功能标记为完成：

- 数据库变更由用户执行并确认。
- 每次终态失败都有当前原因和历史记录。
- 选中重试、全部重试和单条重新标注共享可靠的重试逻辑。
- 重试请求幂等，并发不会重复增加次数或重复占用数据项。
- Worker 自动派发成功；派发失败可以从 `pending` 状态恢复。
- 失败结果不会覆盖已接受、已转人工或人工任务数据。
- Worker、后端和前端自动化检查通过。
- 用户完成真实开发环境联调。
- README、业务文档、设计方案和实现步骤与代码一致。
