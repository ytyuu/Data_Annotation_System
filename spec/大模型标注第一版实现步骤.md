# 大模型标注第一版实现步骤

## 1. 使用方式

本文档是大模型标注第一版的执行计划。后续实现必须按本文档的阶段顺序推进，并在完成每个检查项后更新状态。

状态约定：

- `[ ]`：尚未开始。
- `[~]`：正在进行。
- `[x]`：已经完成并通过本阶段检查。
- `[!]`：遇到阻塞，必须在对应阶段记录原因和解决结论。

执行规则：

1. 开始一个阶段前，确认它依赖的阶段已经完成。
2. 每次只将一个主要阶段标记为 `[~]`。
3. 阶段内代码完成不等于阶段结束，必须满足退出条件。
4. 关键设计变化先更新 [大模型标注第一版设计方案](./大模型标注第一版设计方案.md)。
5. 不跳过数据库并发、幂等、租约和结果校验测试。
6. 不把第一版非目标临时塞入当前实现。

## 4. 阶段 1：数据库结构和环境配置

- Codex 禁止：连接数据库、执行数据库修改命令、查询数据库状态、查看数据库中的结构或数据。

## 6. 阶段 3：提供方批次接口

### 实现任务

- [ ] 实现创建批次请求校验和默认值。
- [ ] 校验数据集所有权、状态、schema 和内容类型。
- [ ] 使用单事务和 `FOR UPDATE SKIP LOCKED` 选择数据项。
- [ ] 将选中数据项更新为 `ai_processing`。
- [ ] 保存 schema、guide、模型和配置快照。
- [ ] 批量创建 AI 结果明细。
- [ ] 实现按数据集查询批次列表。
- [ ] 实现批次详情和精确计数返回。
- [ ] 在没有 pending 数据时返回 409。

### 验证任务

- [ ] 非拥有者不能创建或查看批次。
- [ ] `draft/completed` 数据集不能创建批次。
- [ ] 不支持的 schema 和内容类型返回明确错误。
- [ ] 两个并发创建请求不会选中同一数据项。
- [ ] 事务中任一步骤失败时，数据项状态和批次记录全部回滚。

### 卡点

- API 当前使用 `TRANSACTION_REPEATABLE_READ`，并发冲突需要识别 PostgreSQL 序列化异常并有限重试。
- 大批量 `batchInsert` 的事务大小需要限制，因此第一版 `maxItems` 上限为 5000。

### 退出条件

- 提供方可以创建、列出和查看批次。
- 被 AI 锁定的数据项不会进入人工首次领取池。

## 7. 阶段 4：Worker 领取、租约和结果上传接口

### 实现任务

- [ ] 实现领取 pending 和过期 processing 结果的原子 SQL。
- [ ] 写入五分钟租约并增加 `attempt_count`。
- [ ] 达到最大尝试次数时转为 `failed`。
- [ ] 首次领取时将批次更新为 `running`。
- [ ] 返回批次规则快照和经过 metadata 白名单过滤的数据。
- [ ] 实现成功结果和单条失败的分段上传。
- [ ] 校验批次、resultId、itemId、roundNo 和数据项状态。
- [ ] 实现标准化结果哈希和幂等冲突处理。
- [ ] 实现后端风险分流和稳定抽样。
- [ ] 实现批次级不可恢复错误上报。
- [ ] 在无待领取或处理中结果时将批次置为 `completed`。

### 验证任务

- [ ] 两个领取请求不会获得同一条有效租约。
- [ ] 过期租约可以重新领取，未过期租约不能被抢占。
- [ ] 相同结果重复上传成功，不同结果覆盖返回 409。
- [ ] 低置信、高风险和缺理由结果进入 `needs_review`。
- [ ] JSON 或业务结构错误进入 `failed`。
- [ ] 批次计数与明细实时一致。

### 卡点

- 领取、租约和状态更新必须是一条原子 SQL 或同一锁事务。
- Worker 网络超时后可能不知道上传是否成功，幂等逻辑必须先于 Worker 重试完成。
- `completed` 不能依赖 Worker 主动通知，要以数据库中是否仍有 pending/processing 为准。

### 退出条件

- 不接入真实模型也可以通过接口模拟完整领取和上传过程。
- Worker 崩溃不会造成永久锁死。

- `finalized_by` 应记录执行采用动作的提供方，而不是 Worker 或空值。

### 退出条件

- AI 候选结果可以安全汇入现有最终结果和审核流程。
- 失败、转人工和重跑都不存在悬空数据项。
