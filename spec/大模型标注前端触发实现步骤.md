# 大模型标注前端触发实现步骤

## 1. 使用方式

本文档是大模型标注「前端触发 Worker」能力的执行计划，构建在[大模型标注第一版实现步骤](./大模型标注第一版实现步骤.md)之上。后续实现必须按本文档的阶段顺序推进，并在完成每个检查项后更新状态。

状态约定：

- `[ ]`：尚未开始。
- `[~]`：正在进行。
- `[x]`：已经完成并通过本阶段检查。
- `[!]`：遇到阻塞，必须在对应阶段记录原因和解决结论。

执行规则：

1. 开始一个阶段前，确认它依赖的阶段已经完成。
2. 每次只将一个主要阶段标记为 `[~]`。
3. 阶段内代码完成不等于阶段结束，必须满足退出条件。
4. 关键设计变化先更新本文档的架构与决策章节，再改代码。
5. 不破坏第一版已有的 CLI 手动运行方式，常驻服务与 CLI 并存。
6. 不把本期非目标（重跑、取消、多批次调度队列）临时塞入当前实现。

## 2. 背景与目标

第一版大模型标注是「创建与执行分离」的两段式：提供方在页面只能创建批次（状态 `pending`）、查询和审核结果；真正执行必须有人手动在机器上运行 `pnpm --filter ai-worker dev -- --batch-id <id>`（见第一版阶段 0：明确 Worker 第一版按指定 `batchId` 手动运行）。

本期目标是打通「前端 → JVM 后端 → Node Worker」的触发链路：提供方在页面点击「运行」即可让 Worker 异步执行对应批次，并在页面轮询看到进度推进，无需接触命令行。

### 非目标（留待后续）

- 批次失败后的前端「重跑」。
- 运行中批次的「取消」（Worker 侧运行实例中断与后端 `cancelled` 状态置位）。
- 多批次调度队列、优先级和限流。
- Worker 水平扩容与多实例负载均衡。

## 3. 架构方案（方案 A：Worker 常驻 + 后端派发）

前端不能直接调用 Worker：Worker 无入站 HTTP 服务，且持有 `LLM_API_KEY` 不可暴露给浏览器。因此在中间建立后端派发通道。

链路：

```
浏览器（provider JWT）
  └─ POST /api/provider/ai-annotation-batches/{batchId}/run
       └─ 后端校验归属与状态 → 用 java.net.http.HttpClient 派发
            └─ POST {AI_WORKER_BASE_URL}/run （Bearer WORKER_TRIGGER_TOKEN）
                 └─ Worker 立即返回 202，后台执行既有 BatchRunner
                      └─ Worker 用 AI_WORKER_TOKEN 回调后端 claimItems / uploadResults / failBatch
浏览器运行中轮询 GET .../ai-annotation-batches（getBatch）刷新进度，直到 completed / failed
```

### 关键设计决策（已确认）

- Worker 常驻 HTTP 服务用 Node 内置 `node:http` 实现，零新增依赖，契合 Worker 当前极简风格（仅 `dotenv` / `openai` / `zod`）。
- 本期前端仅提供「运行」一个批次操作，不含重跑与取消。
- 后端 → Worker 的派发鉴权使用**新增**的 `WORKER_TRIGGER_TOKEN`，与 Worker → 后端的 `AI_WORKER_TOKEN` 分开，两条链路职责与轮换独立。
- 触发语义为异步：Worker 立即返回 `202`，实际执行放后台；前端在批次 `running` 期间轮询刷新进度。
- 后端派发成功后**不抢先**将批次改为 `running`，仍沿用第一版「Worker 首次 `claimItems` 时被动转 `running`」的机制；这样 Worker 不可达时批次保持 `pending` 可再次触发，不会卡死在 `running`。
- 同一 `batchId` 在 Worker 侧不允许并发重复执行，重复触发返回 `409`。

### 新增环境变量

| 变量 | 位置 | 说明 |
|---|---|---|
| `WORKER_TRIGGER_TOKEN` | 根 `.env` | 后端派发到 Worker `/run` 的共享密钥，Worker 与后端共读同一 key。 |
| `WORKER_HTTP_PORT` | 根 `.env` | Worker 常驻服务监听端口，默认 `7100`。 |
| `AI_WORKER_BASE_URL` | 根 `.env` | 后端派发时访问的 Worker 基础地址，例如 `http://localhost:7100`。 |

沿用第一版：`AI_WORKER_TOKEN`（Worker 回调后端）、`API_BASE_URL`、`LLM_*`。

### 新增接口

| 方法 | 路径 | 鉴权 | 说明 |
|---|---|---|---|
| `POST` | `/api/provider/ai-annotation-batches/{batchId}/run` | provider JWT | 提供方触发批次执行，成功返回 `202`。 |
| `POST` | `/run`（Worker 侧内部服务） | `Bearer WORKER_TRIGGER_TOKEN` | 后端派发触发，body 含 `batchId` 及可选运行参数，成功返回 `202`。 |
| `GET` | `/health`（Worker 侧内部服务） | 无 | 存活探测。 |
