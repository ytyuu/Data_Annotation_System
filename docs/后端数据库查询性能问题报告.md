# 后端数据库查询性能问题报告

> 检查范围：`apps/api/src/main/kotlin/com/example/api/service/` 下的业务查询代码
> 数据库：PostgreSQL 14
> ORM：Exposed
> 检查日期：2026-06-04

---

## 1. 严重问题：互查任务查询存在大量数据回表

**位置：** `AnnotatorDatasetService.kt` 第 424-471 行（`findReviewableItemRows`）

**问题描述：**

`findReviewableItemRows` 方法为了找出可互查的数据项，执行了**三轮全表扫描**：

1. 第一轮：从 `annotations` 表查出所有 `annotation_type='annotation' AND status='submitted'` 的 `item_id`
2. 第二轮：从 `annotations` 表查出所有 `annotation_type='review'` 的 `item_id`
3. 第三轮：用 `candidateItemIds = originalItemIds - reviewedItemIds - excludedItemIds` 计算差集后，再用 `IN` 查询 `data_items`

**风险：**

- 当 `annotations` 表数据量增大时，前两轮查询会返回大量 `item_id`，内存中做 `Set` 差集效率低
- 第三轮 `DataItemsTable.id inList candidateItemIds` 中 `IN` 列表可能非常长，导致查询性能急剧下降
- 当前 `annotations` 表只有 `idx_annotations_item_type` 索引，但查询条件是 `annotation_type + status`，缺少复合索引

**建议：**

```sql
-- 添加复合索引
CREATE INDEX idx_annotations_type_status ON annotations(annotation_type, status);
```

同时考虑将三轮查询合并为一个 SQL 子查询，用 `NOT EXISTS` 替代内存差集：

```sql
SELECT di.* FROM data_items di
WHERE di.dataset_id IN (?) 
  AND di.status IN ('assigned','annotated','disputed')
  AND di.id NOT IN (?)  -- excludedItemIds
  AND EXISTS (
    SELECT 1 FROM annotations a1 
    WHERE a1.item_id = di.id AND a1.annotation_type = 'annotation' AND a1.status = 'submitted'
  )
  AND NOT EXISTS (
    SELECT 1 FROM annotations a2 
    WHERE a2.item_id = di.id AND a2.annotation_type = 'review'
  )
```

## 2. 中等问题：`listOpenDatasets` 中 `totalActive` 查询在应用层求和

**位置：** `AnnotatorDatasetService.kt` 第 124-130 行

**问题描述：**

```kotlin
val totalActive = AnnotationTaskBatchesTable
    .selectAll()
    .where { ... }
    .sumOf { it[AnnotationTaskBatchesTable.totalCount] }
```

这里先把所有符合条件的记录查出来，再在 Kotlin 内存中求和。虽然 `totalCount` 字段很小，但如果一个标注员有大量历史任务单，仍然会加载不必要的行。

**建议：**

使用 SQL `SUM()` 聚合函数直接在数据库层计算：

```kotlin
val totalActive = AnnotationTaskBatchesTable
    .slice(AnnotationTaskBatchesTable.totalCount.sum())
    .selectAll()
    .where { ... }
    .firstOrNull()?.get(AnnotationTaskBatchesTable.totalCount.sum()) ?: 0
```

## 3. 中等问题：`AnnotationsTable` 缺少 `task_id` 上的索引

**位置：** `Tables.kt` 第 92-113 行

**问题描述：**

`AnnotationsTable` 在 `taskId` 字段上定义了 `uniqueIndex`，但 Exposed 的 `uniqueIndex` 在某些版本下可能不会自动创建普通 B-tree 索引（取决于具体实现）。在 `submitAnnotationBatch` 中频繁使用：

```kotlin
AnnotationsTable.selectAll().where { AnnotationsTable.taskId eq submission.taskId }
```

**建议：**

确认 `task_id` 上是否有索引。如果 `uniqueIndex` 没有创建底层索引，需要显式添加：

```sql
CREATE INDEX idx_annotations_task_id ON annotations(task_id);
```

## 4. 轻微问题：`DatasetQueryHelper.countCompletedItems` 和 `countDisputedItems` 使用 `groupingBy`

**位置：** `DatasetQueryHelper.kt` 第 28-41 行、第 71-84 行

**问题描述：**

```kotlin
return DataItemsTable
    .select(DataItemsTable.datasetId)
    .where { ... }
    .groupingBy { it[DataItemsTable.datasetId] }
    .eachCount()
```

这里 `select(DataItemsTable.datasetId)` 只选了一列，但 `groupingBy` 是在 Kotlin 内存中做分组计数。虽然数据量不大时没问题，但用 SQL `GROUP BY + COUNT(*)` 更高效。

**建议：**

改用 Exposed 的聚合查询：

```kotlin
DataItemsTable
    .slice(DataItemsTable.datasetId, DataItemsTable.datasetId.count())
    .selectAll()
    .where { ... }
    .groupBy(DataItemsTable.datasetId)
    .associate { it[DataItemsTable.datasetId] to it[DataItemsTable.datasetId.count()].toInt() }
```

## 5. 轻微问题：`submitReview` 中多次全量 `count` 查询

**位置：** `ProviderDatasetService.kt` 第 837-855 行

**问题描述：**

在 `submitReview` 中，如果没有传入 `sampledItemCount`，会执行一次 `count`；然后又单独执行一次 `disputedCount`。这是两次全表扫描。

**建议：**

可以合并为一次查询用 `CASE WHEN` 分别计数，不过当前逻辑清晰，数据量不大时影响有限。

## 6. 轻微问题：`finishReview` 中三次独立 `count` 查询

**位置：** `ProviderDatasetService.kt` 第 1005-1030 行

**问题描述：**

```kotlin
val acceptedCount = ...count()
val rejectedCount = ...count()
val totalReviewable = ...count()
```

三次查询条件类似，可以合并为一次 `GROUP BY status` 查询。

**建议：**

合并为单次查询：

```kotlin
DataItemsTable
    .slice(DataItemsTable.status, DataItemsTable.id.count())
    .selectAll()
    .where { datasetId eq ... and status inList ... }
    .groupBy(DataItemsTable.status)
```

## 7. 索引覆盖度检查

根据 `数据标注数据库设计.sql` 中的索引定义，以下高频查询条件**已有索引支持**：

| 索引名 | 表 | 字段 | 状态 |
|---|---|---|---|
| `idx_datasets_provider_status_updated_at` | `datasets` | `provider_id + status + updated_at` |  |
| `idx_data_items_dataset_status` | `data_items` | `dataset_id + status` |  |
| `idx_annotation_tasks_annotator_dataset_status` | `annotation_tasks` | `annotator_id + dataset_id + status` |  |
| `idx_annotation_task_batches_annotator_dataset_status` | `annotation_task_batches` | `annotator_id + dataset_id + status` |  |

**缺少的索引：**

| 表 | 字段/条件 | 用途 |
|---|---|---|
| `annotations` | `(annotation_type, status)` | `findReviewableItemRows` 中频繁查询 |
| `annotations` | `(item_id, annotation_type, status)` | 互查候选筛选 |

## 8. 事务隔离级别

**位置：** `DatabaseFactory.kt` 第 26 行

**问题描述：**

当前使用 `TRANSACTION_REPEATABLE_READ`，在 PostgreSQL 中这会导致：

- 幻读保护开销
- 长时间事务可能阻塞其他操作
- 标注提交场景下并发冲突概率增加

**建议：**

考虑是否可以用 `TRANSACTION_READ_COMMITTED`。数据标注系统的业务逻辑（领取任务、提交标注）本身通过状态机和唯一约束保证一致性，不一定需要 `REPEATABLE_READ`。

## 总结

| 优先级 | 问题 | 影响 |
|---|---|---|
| 高 | `findReviewableItemRows` 三轮查询 + 内存差集 | 数据量大时查询时间指数增长 |
| 中 | `totalActive` 应用层求和 | 不必要的内存开销 |
| 中 | `annotations` 缺少 `(annotation_type, status)` 复合索引 | 全表扫描风险 |
| 低 | `countCompletedItems` 内存分组 | 可用 SQL GROUP BY 优化 |
| 低 | `submitReview` / `finishReview` 多次 count | 可合并查询 |
