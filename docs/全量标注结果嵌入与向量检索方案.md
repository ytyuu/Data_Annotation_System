# 全量已接受标注结果嵌入与向量检索方案

> 版本：v0.2  
> 估算日期：2026-07-17  
> 估算范围：10 万～100 万条评论，仅考虑外部嵌入 API  
> 说明：厂商价格会调整，正式采购前应以控制台、合同和实时配额为准。

## 1. 目标和边界

本方案将去重后的已接受文本标注结果异步调用嵌入 API，保存到 PostgreSQL + pgvector，并在大模型第一轮输出候选分类后检索对应类别的相似正例、近邻反例和边界示例。

本文只评估：

- 10 万和 100 万条短评论。
- 外部嵌入 API。
- API 费用、处理时间和向量存储。

本文不讨论本地开源模型、GPU、自建推理服务和千万级数据。

## 2. 推荐工作流

```text
标注结果最终接受
        ↓
规范化文本并生成内容哈希
        ↓
精确去重，近重复评论合并
        ↓
写入异步嵌入任务
        ↓
批量调用嵌入 API
        ↓
保存向量、模型、维度和版本
        ↓
第一轮大模型输出 1～3 个候选分类
        ↓
按数据集、候选分类和示例资格过滤
        ↓
向量召回约 20 条
        ↓
选择 4～8 条正例、反例和边界示例
        ↓
第二轮逐条复核
```

嵌入任务必须异步执行，不能阻塞标注结果接受事务。任务需要支持幂等、批量调用、重试和失败补偿。

## 3. 全量可检索与可注入示例分离

“已接受”不等于“适合注入 Prompt”。建议分成两个资格字段：

- `searchable`：允许进入向量检索候选集。
- `eligible_for_prompt`：允许作为示例注入大模型。

| 接受来源 | 是否嵌入并可检索 | 是否可直接注入 Prompt |
| --- | --- | --- |
| 人工修改后接受 | 是 | 作为高优先级候选，确认后可注入 |
| 人工逐条确认 | 是 | 可注入 |
| 抽检确认 | 是 | 可注入 |
| 未抽检的批量接受 | 可进入 | 默认不可注入 |
| AI 自动接受 | 可进入 | 默认不可注入 |
| 驳回、失败或争议 | 否 | 否 |

这样可以对接受结果进行全量相似性分析，同时避免错误 AI 结果通过检索形成自我强化。

## 4. 数据与估算假设

统一使用以下假设：

- 去重后评论数：10 万或 100 万。
- 单条评论嵌入输入：30～100 Token。
- 时间估算默认平均 50 Token/条。
- API 价格不考虑免费额度、促销、资源包和企业合同折扣。
- OpenAI 美元价格按 `1 美元 ≈ 7.2 元人民币` 仅作内部换算。
- pgvector `vector` 使用 Float32，每条占用 `4 × dimensions + 8` 字节。
- 向量表加 ANN 索引按原始向量容量的 2～3 倍规划，不包含原评论正文、WAL、备份和数据库副本。

实际 Token 数必须在上线前抽样统计，不能长期使用字符数直接代替 Token 数。

## 7. 全量处理时间

按平均 50 Token/条：

- 10 万条约 500 万 Token。
- 100 万条约 5000 万 Token。

忽略请求排队、RPM、网络和重试时：

```text
处理分钟数 = 总 Token ÷ 实际有效 TPM
```

| 实际有效吞吐 | 10 万条 | 100 万条 |
| ---: | ---: | ---: |
| 10 万 TPM | 50 分钟 | 8.33 小时 |
| 50 万 TPM | 10 分钟 | 1.67 小时 |
| 100 万 TPM | 5 分钟 | 50 分钟 |
| 500 万 TPM | 1 分钟 | 10 分钟 |

“有效 TPM”应使用实际完成 Token 除以端到端时间计算，包含厂商限流、请求批量上限和重试损耗。Batch 接口虽然价格低，但可能有排队时间，不应把理论 TPM 当成交付时间承诺。

假设每月新增：

- 其中 20% 进入困难样本第二轮。
- 每条查询平均 50 Token。

则查询向量用量分别为 100 万和 1000 万 Token/月：

| 实时 API | 10 万新增评论/月 | 100 万新增评论/月 |
| --- | ---: | ---: |
| OpenAI `text-embedding-3-small` | 约 ¥0.14/月 | 约 ¥1.44/月 |
| OpenAI `text-embedding-3-large` | 约 ¥0.94/月 | 约 ¥9.36/月 |
| 阿里云实时 / 百度千帆 / 腾讯云 | 约 ¥0.50/月 | 约 ¥5.00/月 |

查询嵌入费用基本可以忽略，但应将同一模型小批次中的困难评论批量提交，减少网络请求数和延迟。

## 9. 向量存储容量

pgvector Float32 `vector` 的原始容量公式：

```text
单条向量字节数 = 4 × dimensions + 8
```

| 维度 | 10 万条原始向量 | 100 万条原始向量 | 10 万条含索引规划 | 100 万条含索引规划 |
| ---: | ---: | ---: | ---: | ---: |
| 512 | 约 0.21 GB | 约 2.06 GB | 约 0.4～0.6 GB | 约 4.1～6.2 GB |
| 1024 | 约 0.41 GB | 约 4.10 GB | 约 0.8～1.2 GB | 约 8.2～12.3 GB |
| 1536 | 约 0.62 GB | 约 6.15 GB | 约 1.2～1.8 GB | 约 12.3～18.5 GB |

参考：<https://github.com/pgvector/pgvector>

实际数据库空间还需要加入：

- 向量记录元数据。
- 原评论正文或外键。
- PostgreSQL 行和页开销。
- WAL、备份和只读副本。
- 索引重建时的新旧索引并存空间。

如果模型支持自定义维度，第一版建议优先测试 512 和 1024 维。是否降维应由实际正反例召回质量决定。

## 10. 存储金钱估算

不同云数据库和磁盘价格差异很大，因此使用变量计算：

```text
月存储费用
= 含索引容量（GB）× 存储单价（元/GB/月）× 副本和备份系数
```

仅用于内部规划，假设：

- 存储单价：¥0.50/GB/月。
- 暂不计算副本和备份，即系数为 1。

| 维度 | 10 万条月存储费 | 100 万条月存储费 |
| ---: | ---: | ---: |
| 512 | 约 ¥0.20～¥0.30 | 约 ¥2.05～¥3.10 |
| 1024 | 约 ¥0.40～¥0.60 | 约 ¥4.10～¥6.15 |
| 1536 | 约 ¥0.60～¥0.90 | 约 ¥6.15～¥9.25 |

若生产环境使用主库、只读副本和独立备份，实际存储费用通常需要再乘以对应副本与备份系数。上表不是任何云厂商的报价。

## 11. API 选型建议

### 快速验证

- 先抽取 1 万～10 万条人工确认样本。
- 至少测试一个国内 API 和一个国际 API。
- 使用相同维度比较候选标签内的正例、反例和边界示例召回。
- 记录实际 Token、完成时间、限流、失败率和费用。

### 全量回填

- 数据允许外发时优先使用 Batch API，降低价格和请求管理成本。
- 按主键游标分页，批量调用并幂等写入。
- 初始数据写入完成后再统一构建 ANN 索引。
- 记录 `embedding_provider`、`embedding_model`、`dimensions` 和 `embedding_version`。

### 增量生产

- 标注接受事务只写嵌入任务，不同步等待 API。
- Worker 批量消费并自动重试。
- 在线检索只使用 `embedding_status = ready` 的记录。
- API 或模型版本切换时建立新索引，验证后再淘汰旧版本。

## 12. 结论

在 10 万～100 万条短评论范围内：

- 一次性 API 嵌入费用约为几角到几十元，价格不是主要障碍。
- 按平均 50 Token/条，处理时间主要由有效 TPM 和 Batch 排队决定，约为数分钟到数小时。
- 512～1024 维向量连同 ANN 索引，10 万条约占 0.4～1.2 GB，100 万条约占 4.1～12.3 GB。
- 成本决策应优先考虑数据合规、向量召回质量、限流和可复现性，而不是只比较每百万 Token 单价。
- 全量接受结果可以进入检索库，但只有人工确认或抽检通过的高质量结果才应允许注入 Prompt。
